**建立日期**: 2026-05-25  
**類別**: 產品設計 / Kano Must-Do  
**關聯版本**: v2.3.x（建議）

## 一、戰略背景

### 為什麼這是 Kano Must-Do

餐廳收據是一個被嚴重低估的**實體觸點**。現有 POS 收據 = 冷冰冰的清單，用完即丟。但如果收據本身成為一個**驚喜時刻**，顧客就會：

1. 拍照 → 發 Instagram/小紅書
2. 保留收據（記念）
3. 主動告訴朋友「這家店的收據超特別」

**傳播路徑**：有趣收據 → 顧客拍照 → 社群曝光 → 新顧客好奇 → 到店消費  
成本：幾行 ESC/POS 字符，零額外硬件。  
ROI：潛在無限，因為每張打印都是一次零成本投放。

### 與 v2.2.0 自助點單的協同

自助點單完成後，顧客手機上顯示「訂單已送出」頁面——目前是空白確認。  
如果這個頁面也展示**今日運勢**或**本桌專屬詩句**，等待上菜期間的體驗就完全不同了。

## 二、功能模組拆解

### 模組 A：ASCII / BBS 藝術圖塊

熱敏打印機是字符設備，等寬字體、點陣感天然契合 BBS/像素藝術美學。

**實現方式**（兩種，可共存）：

#### A1 — 純字符 ASCII Art（最相容）
```
 ☆ ☆ ☆  LUCKY RECEIPT  ☆ ☆ ☆
     ／￣＼
    （  °▽°）  いただきます！
     ＼＿／
```
- 適用所有熱敏打印機（58mm/80mm）
- 字符集依賴：需要打印機支援 UTF-8 或選用 CP932/CP936（依機型）
- 安全方案：全 ASCII 藝術 + 半形字符

#### A2 — ESC/POS Bitmap 圖像（高質感）
使用 `GS v 0` 指令打印位圖，可做：
- 可愛 logo（布穀鳥圖案）
- 食物像素圖（根據訂單內容動態選圖）
- 每月季節插圖

技術限制：需要事先把圖片轉成 ESC/POS 二值點陣格式（每行8px為一組）；  
58mm 紙 = 384px 寬，80mm 紙 = 576px 寬。

**建議先做 A1（純字符），A2 作為 v3.x 升級項目**。

### 模組 B：運勢 / 抽籤模組

**設計原則**：
- **只有吉，沒有凶** — 小吉、中吉、大吉 三級
- 絕對不出現「凶」「末吉」「小凶」等負面結果
- 心理學基礎：正面強化讓顧客以好心情結束用餐體驗

**隨機種子策略**（可在模板設定中選擇）：

| 策略 | 說明 | 適用場景 |
|------|------|----------|
| `daily` | 全店當天同一籤 | 製造「今日運勢」話題感 |
| `per_table` | 桌號 × 日期 哈希 | 每桌獨特，引發比較討論 |
| `per_order` | 訂單 ID 哈希 | 每張收據完全唯一 |

**建議預設**：`per_table` — 一桌朋友看到相同運勢，有共同話題；  
不同桌不同運勢，引發「你們桌幾吉？」的對話。

---

**運勢文本庫草案**：

```

### 模組 C：詩句 / 語錄塊

**語言庫方向**：

#### C1 — 古典中文詩詞（食物/飲食主題）
精選與「食」「飲」「相聚」相關的詩詞片段：

```
「人間有味是清歡」— 蘇軾
「莫笑農家臘酒渾，豐年留客足雞豚」— 陸游
「春風十里不如你」（改編，適合情侶桌）
「此刻江南正是好，蓴羹鱸膾思鄉時」
「舉杯邀明月，對影成三人」— 李白（搭配飲品訂單）
```

#### C2 — 日語俳句風（簡潔詩意）
適合有日式元素的餐廳，或想要國際感的品牌：

```
一碗温かい
  料理の香りに包まれて
    幸せが満ちる

食べることは
  生きること、愛すること
    感謝していただく
```

#### C3 — 英文 Instagram 風（簡短金句）
最容易被年輕客群截圖分享：

```
"Good food is the foundation of genuine happiness."

"Every meal is a love letter to your body."

#### C4 — 中英混搭（最 INS 感）

## 三、技術實現路徑

### 3.1 模板 JSON 擴充（最小改動）

現有 `elements` 陣列新增三種類型：

```json
// 運勢模組
{
  "type": "fortune",
  "seed_strategy": "per_table",
  "show_level_text": true,
  "show_description": true
}

// 藝術圖塊
{
  "type": "art",
  "variant": "random",
  "category": "food"
}

// 語錄模組
{
  "type": "quote",
  "language": "multilingual",
  "style": "instagram"
}
```

### 3.2 Rust 後端渲染（`print_templates.rs`）

在現有 `render_element()` 函數增加三個 match arm：

```rust
ElementType::Fortune => render_fortune(elem, order_id, table_no, date),
ElementType::Art => render_art_block(elem),
ElementType::Quote => render_quote(elem, seed),
```

關鍵實現細節：

**運勢種子（可重複再現）**：
```rust
let seed = match seed_strategy {
    "per_table" => hash(table_no + date_str),
    "per_order" => hash(order_id),
    "daily"     => hash(date_str),
};
// seed % 100: 0-19 → 小吉, 20-69 → 中吉, 70-99 → 大吉
// 再用 seed / 100 選具體文本
```

**嵌入式文本庫**（編譯進二進制，無需外部文件）：
```rust
static FORTUNE_TEXTS: &[(&str, FortuneLevel, &str)] = &[
    ("大吉", FortuneLevel::DaiJi, "今日萬事俱備..."),
    // ...
];

static QUOTES_ZH: &[&str] = &["人間有味是清歡", ...];
static QUOTES_EN: &[&str] = &["Good food is the foundation...", ...];
```

### 3.3 前端模板編輯器擴充

在 `print-templates-page.tsx` 的 content JSON 編輯器旁邊，新增：
- **元素選擇器面板**：可點選插入 fortune/art/quote 塊
- **運勢預覽**：在 Live Preview 中顯示當前種子的運勢文字

### 3.4 自助點單頁整合（v2.2.0 協同）

在 `self-order-page.tsx` 訂單提交成功後的確認畫面：
- 顯示本桌今日運勢（大字展示）
- 加入「分享運勢」按鈕（Web Share API）
- 附上餐廳名稱 + 品牌 tag

## 四、組合方案建議

根據工作量與病毒傳播效果，建議三個階段：

### 方案一：Quick Win（1-2天工作量）
**目標**：最快上線，驗證效果

- 在現有 `receipt` 模板中加入 `quote` 塊（英文 Instagram 風）
- 純 JSON 配置，後端只需增加字符串查表
- 不需要新增 DB 欄位，只需擴充 Rust render 邏輯

**預期效果**：收據尾部出現一句英文金句，有設計感，客群中年輕人會拍照。

---

### 方案二：Core MVP（3-5天工作量）
**目標**：建立完整的「幸運收據」體驗

- 方案一基礎上 +
- 運勢模組（per_table 種子策略）
- ASCII 藝術圖塊（食物系列，3-5種預設）
- 前端模板編輯器中新增圖形化 element 插入 UI
- 自助點單確認頁顯示運勢

**預期效果**：用戶有話題點（「你們桌幾吉？」），高傳播潛力。

---

### 方案三：Premium（1-2週工作量）
**目標**：打造品牌識別度，成為差異化賣點

- 方案二基礎上 +
- ESC/POS 位圖圖像支援（品牌 logo + 季節插圖）
- 顧客「分享運勢」頁面（Web + QR code 導回餐廳自助頁）
- 多語言語錄庫（中/日/英三選一或混合）
- 依訂單內容動態選圖（有飲品 → 咖啡圖，有甜點 → 蛋糕圖）

## 五、優先級整合建議

目前 v2.2.0 剩餘高優先項（T4.1 production bundle 驗證）之後，建議：

```
T4.1 驗證完成
    ↓
方案一 Quick Win（語錄塊，後端 + 模板 JSON）
    ↓
同時進行：自助點單頁運勢展示（前端，不需後端）
    ↓
方案二 Core MVP（運勢 + ASCII 藝術）
```

方案一 + 自助點單整合可以**不阻塞其他功能開發**，因為：
- 純後端 match arm 擴充，不影響現有渲染邏輯
- 前端只是新增 element 類型的渲染顯示

## 六、潛在風險與注意事項

| 風險 | 說明 | 對策 |
|------|------|------|
| 打印機字符集 | 非 ASCII 字符在部分打印機顯示為亂碼 | 提供 ASCII-only fallback，模板設定中可選字符集 |
| 文本長度 | 58mm 紙只有 ~24 字寬（英文）/ ~12 漢字 | 按紙寬自動截斷或換行，fortune 文本長度控制在 20 字以內 |
| 每次打印結果不同 | `per_order` 種子策略讓每張收據都不同，廚房單和收據可能不同 | 廚房單不需要運勢，只在 `receipt` 類型模板中啟用 |
| 文化適應性 | 語錄/運勢的文化接受度因客群而異 | 模板完全由店家自配，不強制啟用，預設不包含創意模組 |
