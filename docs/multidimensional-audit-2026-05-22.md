## 1. 審計基線與證據

### 1.1 當前產品定位

- 本產品是本地優先的餐飲桌面作業系統，覆蓋 POS、KDS、庫存、採購、生產、報表、打印、會員、局域網同步。
- 代碼形態是「單體桌面業務系統」而非「微服務」；優勢是交付快，代價是耦合上升後局部演進成本快速增加。

## 2. 執行摘要

### 2.1 總判斷

- **商業面**：方向是對的。對中小餐飲門店來說，「本地優先 + 配方驅動 + 庫存閉環 + 打印/KDS」是有價值的完整作業鏈。
- **安全面**：目前更像「可信門店內網工具」，不是「有嚴格內控與敵手模型的商業軟體」。
- **架構面**：已進入可持續交付與可持續維護的分水嶺。再疊新功能會先壓垮 `App.tsx`、`useAppActions.ts`、`database.rs`、`commands.rs`。
- **市場面**：功能廣度不差，但護城河主要在整合與落地，不在技術壁壘。若沒有更強的多店/協同/運營能力，容易停留在「能用的單店工具」。

### 2.2 最值得優先關注的四件事

1. **權限模型只在前端，不在命令邊界**。這限制了產品對「員工分權」的真實承諾。
2. **局域網同步沒有認證、簽名、版本協商**。它更像便利同步，不是可信同步。
3. **全量 `loadData()` + 單 Mutex SQLite** 已形成結構性性能上限。
4. **核心業務邏輯散在巨型檔案中**，會讓任何進一步的修 bug、擴功能、寫測試都越來越貴。

## 3. 模塊 / 組件 / Spec 審計

### 3.1 POS / Orders / KDS

| 維度 | 判讀 |
|---|---|
| 商業價值 | 這是成交主鏈路，必須快、穩、低認知負擔。 |
| 當前優勢 | 訂單、KDS、出餐、退款、打印已形成基本閉環。 |
| 主要風險 | `App.tsx` 同時負責路由、輪詢、遙測、角色 gating、頁面裝配；訂單頁還直接在 route prop 中做額外 `invoke` 聚合，擴展成本高，見 `src/App.tsx:386-387`。 |
| 對抗視角 | 惡意店員或誤操作更可能造成流程錯誤，而不是外部攻擊；退款、取消、補打、出餐這些操作應被視為審計事件。 |
| Debug 重點 | 先查 `App` 層事件流，再查對應 command，最後查 `database.rs` 實際狀態轉移。 |

### 3.2 Recipes / Menu

| 維度 | 判讀 |
|---|---|
| 商業價值 | 配方是庫存扣減、成本核算、毛利分析的根。 |
| 當前優勢 | 產品已從「菜單管理」進化到「配方驅動作業系統」，這是核心差異點。 |
| 主要風險 | 配方與菜單關係、待完善配方、成本重算、子配方遞迴等邏輯跨前後端分散，變更時容易破壞成本一致性。 |
| Kano | 屬於從 One-dimensional 向 Attractive 過渡的能力，做得越好越能拉開與普通 POS 的差距。 |
| Debug 重點 | 出問題時不要先查 UI；先確認 recipe graph、成本計算、庫存交易是否一致。 |

### 3.3 Inventory / Stocktake / Production / Purchase

| 維度 | 判讀 |
|---|---|
| 商業價值 | 這是毛利、損耗、備料準確性的護城河。 |
| 當前優勢 | FIFO/FEFO、盤點、採購、批次、生產單已覆蓋到可運營層級。 |
| 主要風險 | 業務規則高度依賴資料正確性，一旦 transaction、成本、狀態機任何一層失真，報表就會失真。 |
| 技術債 | `database.rs` 集中承擔庫存、生產、採購、報表，導致測試與重構邊界模糊。 |
| Debug 重點 | 先看 `inventory_txns` 與批次，再看 UI 顯示；不要反過來。 |

### 3.4 Customers / Loyalty

| 維度 | 判讀 |
|---|---|
| 商業價值 | 有助復購與營銷，但不是當前系統的護城河核心。 |
| 市場視角 | 會員能力屬「補齊型」而非「決定購買型」，在單店市場能加分，在連鎖市場不夠。 |
| 主要風險 | 若沒有與支付、券、積分兌換、活動分析形成完整閉環，容易成為維護成本高、營收貢獻低的模塊。 |
| 建議 | 保持最小可用，不要讓它優先擠占庫存/同步/權限等基礎能力的工程資源。 |

### 3.5 Printing

| 維度 | 判讀 |
|---|---|
| 商業價值 | 打印是餐飲軟體的可信度體感來源之一。 |
| 當前優勢 | 已有背景列印、結果事件、LAN/飛鵝雲多路徑。 |
| 安全面 | TSPL 注入已補，但打印仍屬外設高不確定環節。 |
| 主要風險 | 外設網路、驅動、模板資料來源、異常重試與人工補單流程仍可能造成營運摩擦。 |
| Debug 重點 | 先區分模板資料問題、命令生成問題、傳輸問題、打印機狀態問題。 |

### 3.6 Sync / Update / Telemetry

| 維度 | 判讀 |
|---|---|
| 商業價值 | 它們不是主賣點，但直接影響多設備可用性、升級效率與運維能力。 |
| 主要風險 | 同步服務對外綁 `0.0.0.0`，回應 `Access-Control-Allow-Origin: *`，且未見認證/簽名/租戶隔離，見 `src-tauri/src/sync_server.rs:25-27`, `91-97`。 |
| 安全面 | 更新 URL 已做白名單，但更新源仍單點依賴 GitHub Release，見 `src-tauri/src/commands.rs:1932-1939`, `src-tauri/src/updater_check.rs:31-75`。 |
| 對抗面 | 內網其他主機、惡意代理、錯配置 URL、偽造同步源，都是更現實的風險，而不是高級外部攻擊。 |
| Debug 重點 | 區分「連不上」「取到髒資料」「版本不兼容」「靜默失敗」四種故障。 |

## 4. 商業 / 業務性分析

### 4.1 產品真正賣的是什麼

- 不是單純 POS。
- 真正可賣點是「配方驅動的單店餐飲作業閉環」。
- 若對外表述成「多角色安全協同系統」或「多店級同步平台」，現有實作會高估產品能力。

### 4.2 對象市場

| 市場段 | 適配度 | 原因 |
|---|---|---|
| 單店 / 小團隊烘焙、輕餐飲 | 高 | 本地優先、操作鏈完整、導入成本低。 |
| 兩三家分店的老闆直管模式 | 中 | 能用，但同步、權限、審計仍偏弱。 |
| 多店連鎖、強內控 | 低 | 缺後端授權、中心化治理、可追責操作審計。 |

### 4.3 業務風險排序

1. **把前端角色切換誤當後端權限控制去賣**。
2. **把局域網同步誤當強一致同步去承諾**。
3. **在配方與庫存邏輯已重耦合時，過早加更多營銷功能**。
4. **在沒有更細粒度觀測能力前，持續擴大打印與外設組合**。

## 5. 安全性與對抗性分析

### 5.1 安全現狀

| 類別 | 現狀 | 評價 |
|---|---|---|
| 本地角色控制 | 前端 `localStorage` PIN + UI gating | 只適合低對抗場景 |
| IPC 授權 | 未見命令級角色校驗 | 核心缺口 |
| 局域網同步 | 無認證、無簽名、無 TLS、回應全開 CORS | 便利性優先 |
| 更新下載 | 已加 GitHub Release URL 白名單 | 比之前安全，但仍非完整供應鏈保護 |
| 遙測 | 可外送錯誤與運營摘要 | 需要明確資料治理邊界 |
| 打印輸入 | TSPL 已做基礎轉義 | 方向正確 |

### 5.2 對手模型

| 對手 | 可能行為 | 當前暴露 |
|---|---|---|
| 誤操作店員 | 切換角色、誤退款、誤刪資料 | 高 |
| 惡意內部人員 | 改 localStorage、直接操作 UI/IPC | 中高 |
| 同網段設備 | 訪問同步端口、抓取訂單 | 中 |
| 惡意更新/代理 | 干擾 GitHub API 或下載鏈路 | 低到中 |
| 外部互聯網攻擊者 | 遠程直接攻擊桌面端 | 低，前提是無額外暴露 |

### 5.3 關鍵結論

- **這不是「零信任」系統。**
- **它目前的安全邊界是：可信設備 + 可信店員 + 可信內網。**
- 如果未來要把它做成多人協作產品，第一個工程主題不應該是新功能，而應該是：
  - 命令級授權
  - 審計日誌
  - 同步認證
  - 資料來源簽名/版本協商

## 6. Kano 模型

### 6.1 Must-be

- 穩定下單、出餐、收款、取消、打印
- 基礎庫存準確
- 崩潰可恢復、錯誤可追蹤
- 備份可用

### 6.2 One-dimensional

- 報表準確度
- 配方成本準確度
- KDS 響應速度
- 批量操作與載入速度

### 6.3 Attractive

- 多層配方、半成品、批次追蹤
- 多設備同步
- 會員與積分
- 預警與營運摘要

### 6.4 目前判讀

- Must-be 基本成熟，但「權限真實性」「同步可信性」仍不達企業級 Must-be。
- Attractive 功能不少，但部分已開始反向推高維護成本。

## 7. SWOT 模型

| 類別 | 內容 |
|---|---|
| Strengths | 本地優先、配方驅動、餐飲流程覆蓋深、外設整合實用。 |
| Weaknesses | 權限在前端、同步薄弱、巨型檔案、狀態/資料邊界不清。 |
| Opportunities | 面向烘焙/輕餐飲做垂直化模板、將庫存/成本準確性做成核心賣點。 |
| Threats | 通用 POS SaaS 功能追平、同價位競品做出更強雲協同、現有代碼複雜度拖慢迭代。 |

## 8. 市場角度

### 8.1 有機會形成差異化的點

- 配方作為成本與庫存的一級模型，而不是附屬資料。
- 生產單、採購單、批次、KDS、打印都在一個桌面流程內閉環。

### 8.2 不足以形成強差異化的點

- 單純會員、報表、角色切換。
- 沒有後端真授權的角色系統，對成熟商家說服力有限。
- 沒有中心化營運、跨店治理、數據對賬能力，難打多店市場。

### 8.3 建議市場表述

- 適合表述為：「中小餐飲門店的本地優先作業系統」。
- 不宜過度表述為：「高安全多門店協同平台」。

## 9. 技術棧與技術債

### 9.1 技術棧判讀

| 層 | 技術 | 判讀 |
|---|---|---|
| 桌面殼 | Tauri 2 + Rust | 成本/性能比不錯，適合本地工具。 |
| 前端 | React 18 + TS | 足夠，但目前狀態管理偏手工 orchestrate。 |
| DB | SQLite + rusqlite | 單店本地很好，但單 Mutex 限制擴展性。 |
| UI | shadcn/ui + Tailwind | 開發效率高，視覺一致性可控。 |
| 同步 | 自建 TCP/HTTP | 起步快，但長期風險大。 |

### 9.2 技術債清單

| 債務 | 證據 | 影響 |
|---|---|---|
| God File: `database.rs` | 6506 行 | 難拆測、難定位副作用、知識集中。 |
| God File: `commands.rs` | 1993 行 | IPC 邊界不清、授權與驗證難收口。 |
| God Hook: `useAppActions.ts` | 803 行 | 幾乎所有 mutation 都在同一層處理。 |
| God Component: `App.tsx` | 421 行 | 路由、同步、遙測、角色、頁面裝配混在一起。 |
| 全局 reload 模式 | `loadData()` 74 次 | 性能與狀態一致性成本高。 |
| 單連接 Mutex | 189 次 `lock().unwrap()` | 並發讀寫上限明顯。 |
| 測試偏 UI 基礎件 | 測試主要在 `src/components/ui/*.test.tsx` | 業務回歸保護不足。 |

### 9.3 這些技術債的真實代價

- 不是「代碼不漂亮」而已。
- 它們直接提高以下成本：
  - 修一個 bug 需要讀更多上下文
  - 新功能更容易撞壞舊邏輯
  - 無法快速給出高信心測試
  - 很難把權限、審計、同步等橫切能力統一落地

## 10. 模塊化 / 組件化建議

### 10.1 前端

- 把 `useAppActions.ts` 按 bounded context 拆為：
  - `useMenuActions`
  - `useRecipeActions`
  - `useOrderActions`
  - `useInventoryActions`
  - `useOpsActions`（打印/同步/更新/遙測）
- 把 `useAppData.ts` 的全量載入改成：
  - 啟動必要資料
  - 頁面懶載資料
  - mutation 後局部刷新
- 把 `App.tsx` 拆為：
  - `AppShell`
  - `AppTelemetry`
  - `AppSyncClient`
  - `AppRoutes`
  - `AppRoleGate`

### 10.2 後端

- 把 `database.rs` 先按業務域拆，不要按 CRUD 拆：
  - `db/menu.rs`
  - `db/recipes.rs`
  - `db/orders.rs`
  - `db/inventory.rs`
  - `db/purchasing.rs`
  - `db/reporting.rs`
  - `db/printing.rs`
- 把 `commands.rs` 按 IPC namespace 拆，讓授權、驗證、審計能掛在命令群組邊界。

## 11. 以多種立場看同一套系統

| 立場 | 真正在乎什麼 | 當前結論 |
|---|---|---|
| 老闆 | 賣得出、算得清、少出錯 | 系統價值成立，但需要更強審計與權限真實性。 |
| 收銀 | 快、別卡、退款別踩坑 | 主鏈路可用，但全量 reload 會成為體感問題。 |
| 廚師 | 票要準、出餐別亂 | KDS 可用，但同步與狀態一致性要持續盯。 |
| 倉管 | 批次、成本、盤點要可信 | 是目前產品最有價值也最脆弱的區域。 |
| 開發者 | 好改、好測、好定位 | 當前結構已超過舒服區，需要逐步分層。 |
| 攻擊者 / 惡意員工 | 怎麼繞過限制、拿到資料 | 角色與同步是最先被打的點。 |

## 12. 優先級建議

### P0：如果要提升產品可信度，先做這些

1. 命令級授權守衛，不再只依賴前端角色判斷。
2. 同步協議增加共享密鑰、時間戳、版本協商、最小資料集。
3. 為退款、取消、出餐、盤點完成、採購入庫建立可追責審計日誌。

### P1：如果要降低開發摩擦，做這些

1. 拆 `useAppActions.ts`。
2. 把 mutation 後的 `loadData()` 改為局部刷新。
3. 先把 `database.rs` 拆成 inventory / orders / recipes 三個核心模塊。

### P2：如果要提升市場競爭力，做這些

1. 把成本/損耗/毛利做成可營運決策的報表，而不只是靜態圖表。
2. 強化單店到雙店的小規模協作，而不是直接跳多店 SaaS。
3. 用垂直場景模板放大配方驅動優勢，例如烘焙、飲品、中央廚房半成品。

## 13. Debug 參考

### 13.1 出問題先看哪裡

| 症狀 | 先查 |
|---|---|
| UI 卡頓 / 載入慢 | `useAppData.ts` 全量並行拉取、`useAppActions.ts` reload、SQLite lock |
| 同步錯亂 | `App.tsx:255-281`, `commands.rs:1981-1993`, `sync_server.rs` |
| 權限看得到但不可信 | `roles.ts`, `RoleSwitchDialog`, 對應 Tauri command 是否無授權 |
| 庫存 / 成本不準 | `inventory_txns`、batch、recipe 展開、報表 SQL |
| 更新失敗 | `updater_check.rs`, GitHub API 響應, 下載事件 |
| 崩潰 / 黑白屏 | `ErrorBoundary`, `appLogger`, `docs/debug-pipeline.md` |

### 13.2 審計結論的使用方式

- 產品決策時，用第 4、6、7、8 章。
- 安全與內控討論時，用第 5、11、12 章。
- 技術重構排期時，用第 9、10、12 章。
- 排查故障時，用第 13 章，再配合 `docs/debug-pipeline.md`。

## 14. 結論

- Cuckoo 已經不是「原型」，而是有真實營運價值的單店餐飲作業系統。
- 但它也已經不再適合用「單體快速堆功能」的方式繼續長大。
- 下一階段最重要的不是再多做幾個頁面，而是把：
  - 權限真實性
  - 同步可信性
  - 模塊邊界
  - 局部刷新與測試保護
 這四件事做成新的工程底盤。

如果這四件事不先處理，後續任何新功能都會以更高成本疊在更脆弱的地基上。
