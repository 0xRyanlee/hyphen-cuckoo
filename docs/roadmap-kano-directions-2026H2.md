## 方向四:魅力增量(A)— 克制選做

A 層是本品差異化,但邊際遞減,**避免堆砌冷門組件(防 I/R)**。

| # | 項 | Kano | 要點 |
|---|---|---|---|
| **4.0** | **集字兌換閉環(截圖持有+掃碼集齊核銷)** | **A** | **無會員可實現**,把已交付的集字組件升級為完整兌換玩法。詳見下方「合理性與方案」。**P1 候選** |
| 4.1 | 節日限定營銷皮膚 | A | 春節/中秋等彈窗/台卡主題,季節性驚喜 |
| 4.2 | 集字「換裝」字組 | A | 店主自定義字組(如店名),提升參與感(可與 4.0 同期) |
| 4.3 | 運勢/詩句內容擴充 | A | 擴充語料庫,降低重複感 |
| 4.4 | 海報模板多樣式 | A | 台卡/活動海報多套版式可選 |

**原則**:4.1–4.4 是「現有機制換皮/擴容」,低成本高感知;4.0 是有實質價值的新閉環。

### 4.0 集字兌換閉環 — 合理性與方案(修正先前「需會員」誤判)

**先前誤判**:曾認為集字兌換需「追蹤某用戶集了幾個字」→ 需會員 → 封存。

**修正洞察(用戶提出)**:**集字遊戲無需系統追蹤進度**。字的「持有」由顧客用**截圖**自持(分散、離線),系統保持無狀態;集齊後顧客把 N 張截圖發微信給老闆或到店出示,老闆掃碼/輸序號核銷即可。完全契合既有「純截圖、無會員」模型。

**完整鏈路**
```
顾客每单下单 → 弹窗给【本单之字】+ 该字的核销码(qrcode) + 短序号
  → 顾客截图保存(自持，系统不追踪)
集齐指定字组(如 恭喜发财) → 微信发 N 张截图给老板 / 到店出示
店员「集字兑换」模式：依次扫码 or 输序号
  → 系统累积显示已收集的字(恭✓ 喜✓ 发✓ 财✓)
  → 凑齐字组 → 「确认兑换」→ 批量销掉这 N 张码(void) + 记一次集字兑换完成
```

**為何無會員仍成立**
- **持有**:截圖即憑證,用戶自己保管,系統無需賬號存進度。
- **防偽**:每張字碼 HMAC 簽名(已有)+ 一次性 void(已有)→ 同一截圖不能重複用、不能偽造。
- **集齊判斷**:老闆掃碼累積,系統按字組驗證是否湊齊(後端校驗 N 張屬於同一活動且覆蓋全部字);肉眼+系統雙重。
- **後備通道**:截圖含短序號,掃碼失敗時老闆手輸序號核銷(對齊已有 D3 核銷碼入口)。
- **微信流轉**:截圖天然可微信發送,老闆收圖後掃屏/輸號,符合用戶描述的真實場景。

**所需開發(P1 候選)**
| 子項 | 要點 |
|---|---|
| a. 集字券截圖加序號 | 彈窗集字卡顯示短序號(token 短碼/order_no),供手輸後備 |
| b. 集字活動定義 | 店主設定字組(如「恭喜發財」)+ 獎品;字碼綁定活動 |
| c. 老闆「集字兌換」模式 | 累積掃碼/輸號 → 顯示已集字 → 湊齊校驗 → 批量 void + 記兌換 |
| d. 後端集齊校驗 | 校驗 N 張碼簽名有效、未 void、屬同活動、覆蓋全字組 |

**复用**:字碼簽發/核銷復用現有 `marketing_qr_tokens` + `redeem_marketing_qr_token`;只新增「活動字組定義」與「批量集齊核銷」語義。

## 方向五:戰略邊界(A,含 R 風險)— 封存待拍板

突破「無會員 / 本地優先」邊界,需先做產品決策再動工。

| # | 項 | Kano | 風險 | 觸發條件 |
|---|---|---|---|---|
| 5.1 | 雲端多店(E4) | A | 強加雲依賴給單店=R | 出現連鎖客戶 |
| 5.2 | 輕量身份(手機/openid)→ 雙向裂變 | A | 登入摩擦=R | 接受身份系統後 |

**判斷**:兩項均誘人(A),但對核心「單店無會員」用戶是反向摩擦(R)。**不應為了功能而犧牲現有用戶的零摩擦體驗**;待戰略客戶出現再評估。
(原 5.3 跨單集字已修正為方向四 4.0,無需身份即可實現,移出封存。)

## 方向六:不做清單(R / I)— 明確規避

| 項 | 類 | 理由 |
|---|---|---|
| 強制會員系統 | **R** | 用戶明確拒絕,做了即降滿意 |
| 無條件強制核銷 PIN | **R** | 增摩擦,已做成可選(未設則放行) |
| 繁瑣多步配置 | **R** | 違背防呆零配置定位 |
| health_check 結構化(E3) | I | 用戶無感,前端依賴 "ok" 字串,收益低 |
| CSP unsafe-inline 收斂(E2) | I | Tailwind 依賴,移除成本高、用戶無感 |

## 執行建議(下一版 v3.3 候選)

**v3.3 = 掃碼安全收口 + 集字兌換 + 商城體驗(M/O/A)**:
1. **集字兌換閉環(4.0)** — A 級高感知,把已交付集字組件升級為完整玩法(截圖持有+掃碼集齊核銷)
2. C2 舊碼硬切開關(1.2)
3. campaign 多領限制(1.3)
4. 完整 modifier(2.1)
5. 路由級 lazy-loading(2.2)

**v3.4 = 營銷 ROI**:折價券金額(3.1)+ campaign 效果看板(3.2)

**選做**:方向四 4.1–4.4 魅力換皮/擴容(可穿插)

**不排期**:方向五(待戰略決策)、方向六(規避/暫緩)

## 一句話

未來方向的紀律是 **Kano 排序**:M 級安全(方向一,SEC1 已落地)收口,A 級的**集字兌換閉環(4.0)**——經修正確認**無需會員、純截圖即可實現**——升格為 v3.3 亮點;O 級體驗(方向二三)為 v3.3/v3.4 主線;戰略邊界(方向五)封存待拍板;R/I 級(方向六)明確不做——**守住「單店、無會員、本地、零配置」的產品邊界就是最大的方向**。
