> 版本基線:v2.7.1(2026-06-02)
> 範圍:全用戶流程審計、未來開發方向歸納、里程碑規劃

## 0. 執行摘要

Cuckoo 已具備完整的「堂食掃碼點單 + 收據/彈窗營銷 + 截圖兌獎」骨架,後端營銷組件齊全(13 種),掃碼商城頁(`self-order-page.tsx`)體驗接近 UberEats。但三個方向存在明確缺口,且恰好是本輪 `/goal` 的核心訴求:

| 方向 | 現狀 | 缺口 | 優先級 |
|---|---|---|---|
| **二維碼體系** | 純黑白裸碼,桌號碼/營銷碼皆靜態無區分,無設計、桌號不入圖、營銷碼無核銷廢止 | 桌號碼設計感台卡(印刷貼紙)+ 訂單營銷碼每單唯一可核銷 + 防刷單 | P0 |
| **掃碼後商城頁** | 類 UberEats,功能完整度約 70% | 營銷組件前後端不對齊、無加料、無搜索/推薦、無桌號校驗 | P0/P1 |
| **營銷閉環與數據** | 配置 + 截圖兌獎核銷可用 | 無掃碼/轉化/兌獎數據、無裂變激勵、折價券無金額統計 | P1 |

**最緊迫的技術債(安全)**:`create_self_order` 為 public endpoint,僅校驗 `table_no.len() ≤ 32`,不驗證桌號真實性、無簽名 token。任何人知道 `http://{IP}:9001/#/table/任意字串` 即可下單。動態二維碼的 token 機制將同時解決此安全問題,因此列為 P0。

## 2. 方向 A:二維碼體系(P0)

> **關鍵澄清(用戶決策)**:二維碼分兩大類,機制截然不同。「每次不同」指的是**小票/訂單上的營銷二維碼**(集字類每張唯一、核銷即廢),**不是桌號碼**。桌號碼是**固定**的——要生成高質量圖片供客戶拿去印刷店印成實體貼紙。設計語言對標**微信/支付寶支付碼**。

### 2.1 二維碼分類體系

| 類型 | 載體 | 是否變化 | 防護 | 用途 |
|---|---|---|---|---|
| **桌號碼** | 實體貼紙(印刷店印製) | **固定永久** | 簽名綁桌 + pending 閘門 | 堂食掃碼進商城點單 |
| **訂單營銷碼·集字類** | 小票打印 / 下單彈窗 | **每單唯一** | 一次性 token,核銷即廢 | 集字兌獎,掃碼核銷後失效 |
| **訂單營銷碼·加群類** | 小票打印 / 彈窗 | 固定(可手動換) | 無需廢止 | 掃碼加群/關注,長期有效 |

### 2.2 現狀

`tables-page.tsx` 用 `QRCodeCanvas value={tableUrl} size=200 level="M"` 渲染:
- 純黑白標準碼,白底黑塊,無 logo、無框、無配色(遠未達微信/支付寶設計語言)
- URL `http://{IP}:9001/#/table/{tableNo}` 靜態,桌號僅在 dialog 標題、**不在圖片內**
- 下載的 PNG 是裸碼,不能直接交印刷店做貼紙
- 後端 `print.rs` 有 `qr_code` element,但小票營銷碼**無唯一性、無核銷廢止機制**

### 2.3 桌號碼:固定 + 簽名綁定 + pending 閘門

桌號碼要印成實體貼紙,故**不能用輪換 token**(實體碼無法每日更換)。安全靠兩道:

1. **固定簽名綁定**:URL 帶一個**固定**簽名 token,綁定桌號。`token = base64url({table:"A3", kind:"dine_in"}) + "." + hmac_sha256(...)`。防止偽造任意 `/#/table/隨便填`——後端驗簽 + 校驗桌號存在於 `restaurant_tables`。同一桌貼紙的 token 永久不變,可反覆印刷。
2. **pending 閘門**:自助單默認 `pending`,需店員在 POS/KDS 確認才入廚房(現狀已是此流轉)。即便有人偽造下單,也只生成待確認單,店員一眼可棄,危害可控。

**後端**
- secret 存於 `role_auth_path` 同級新文件(沿用本地文件存儲慣例,不入 git)
- 加 `hmac` + `sha2` crate;新 public endpoint `/api/resolve_table_token` → 驗簽 + 桌號存在性 → `{ table_no, valid }`
- `create_self_order` 要求合法 token 且 `payload.table == 提交 table_no`,堵住「猜 URL 刷單」

### 2.4 訂單營銷碼:每單唯一 + 核銷廢止(「每次不同」核心)

集字兌獎等營銷碼,**每張小票/每次下單彈窗生成唯一 token**——即使抽到同一個字,二維碼也不同。掃碼核銷後該碼失效,防止一碼多用、截圖轉發濫用。

- token payload:`{ order_id, component:"character_collect", char:"喜", nonce, issued }`,HMAC 簽名
- 新增 `marketing_qr_tokens` 表:`token / order_id / component / redeemed_at / void`。掃碼核銷寫 `redeemed_at` + `void=1`,二次掃碼提示「已核銷」
- 與既有 `marketing_redemptions` 打通:核銷碼 = 寫兌獎記錄
- **加群類**:固定 URL(店主在營銷中心填寫的群/公眾號連結),無 token、不廢止,僅店主手動改連結時才變

**這同時升級了截圖兌獎模型**:從「人工核對截圖」進化到「掃碼自動核銷」,店員 PAD 掃小票/彈窗上的營銷碼即完成核銷,免手輸訂單號(呼應 §1.2 店員痛點)。

### 2.5 視覺:微信/支付寶級設計語言

**目標**:中心圓形 logo、品牌色、圓角碼點(rounded modules)、品牌化定位點(custom eye)、簡潔留白——即微信/支付寶支付碼的視覺語言。

**技術選型(用戶定調:對標微信/支付寶)**
| 方案 | 能力 | 結論 |
|---|---|---|
| `qrcode.react` v4 | 僅中心 logo + 純色,**碼點/eye 無法圓角化** | ❌ 達不到目標設計語言 |
| **`qr-code-styling`** | 圓角碼點、漸變、自定義 eye、中心 logo、導出 PNG/SVG | ✅ **採用**,~30KB,評估後納入 bundle |

**桌號台卡**:`qr-code-styling` 生成設計感碼 + React 台卡排版(店名、logo、桌號大字、slogan、「掃一掃 自助點餐」引導),導出**高分辨率 PNG/SVG** 供印刷店製作貼紙(需注意印刷用高 DPI / 矢量 SVG)。

**訂單營銷碼**:小票走 ESC/POS 原生 QR 指令(`print.rs` 已有),彈窗走 `qr-code-styling`;兩者 token 一致,設計感在彈窗呈現。

## 3. 方向 B:掃碼後商城頁面增強(P0/P1)

### 3.1 對齊前後端營銷組件(P0,低成本高價值)

**問題**:`MarketingCard`(`self-order-page.tsx:266`)只渲染 7 種,後端 `print.rs` 支持 13 種。顧客手機端**看不到** `discount_coupon`(折價券——最直接的促銷)、`qr_code`、`product_spotlight`(新品)、`rich_text`、`dish_easter_egg`。

**方案**:複用後端 `render_template_content_preview` 已輸出的 HTML,改 `MarketingCard` 為「優先用後端渲染 HTML(`DOMPurify` 消毒)+ 前端 djb2 種子組件兜底」,或直接補齊 6 種前端渲染。建議前者:單一渲染源,前後端永久對齊。

### 3.2 商城功能補全(P1)

| 項 | 說明 |
|---|---|
| 加料/去料(modifier) | 對齊 POS 能力,顧客自助定制(少冰/加料/去蔥) |
| 規格 sheet 數量選擇 | 一次加多份 |
| 搜索 + 熱銷/推薦標記 | 菜品多時提效;熱銷可由銷量驅動 |
| 套餐/組合 | 提客單價 |
| 掃碼首屏營銷鉤子 | 進店即給營銷觸點(今日運勢/首掃領券),而非僅下單後 |

### 3.3 健壯性

- `create_self_order` 校驗:菜品/規格存在性、`is_available`、qty 上限(後端已部分校驗,需確認規格與下架攔截)
- token 校驗失敗的友善頁面(過期碼引導重掃)

## 5. 技術可行性與架構影響

| 改動 | 影響面 | 成本 | 風險 |
|---|---|---|---|
| HMAC token 簽發/驗證 | 新增 `hmac`+`sha2`;`web_server.rs` 新 endpoint;`tables-page` 改 URL 生成 | 中 | 低(成熟方案) |
| `create_self_order` 強制 token | 改 public endpoint 校驗 | 中 | **中**:需保證舊碼平滑過渡(寬限期/雙模) |
| 品牌台卡二維碼 | `tables-page.tsx` UI + 可選新依賴 | 中 | 低 |
| 營銷組件前後端對齊 | `MarketingCard` 改用後端 HTML | 低 | 低 |
| 商城加料/搜索 | `self-order-page.tsx` + public menu 補 modifier | 中 | 低 |
| 數據分析 | `qr_tokens` 表 + 聚合查詢 + 報表頁 | 中高 | 中(DB 查詢成本,H 維度) |

**架構一致性(I)**:token secret 沿用本地文件存儲慣例;新表走 `schema.rs` migration;新 endpoint 走 `web_server.rs` public 區;前端走 `transport.ts`。無需引入新架構。

**防呆/小店約束**:所有新功能須保持「默認可用、零配置起步」。動態二維碼默認用「每日輪換」,店主無需理解 token;台卡一鍵生成下載;token 過期自動引導重掃。

## 6. 開發里程碑建議

### v2.8.0 — 二維碼體系 + 品牌台卡 + 安全(P0)
- HMAC 簽名(`hmac`+`sha2`),secret 本地文件;沿用本地存儲慣例
- **桌號碼**:固定簽名 token 綁桌;`/api/resolve_table_token` 驗簽 + 桌號存在性;`create_self_order` 要求合法 token(寬限期雙模:舊靜態 URL 暫仍可用 + 提示重新生成台卡)
- **品牌台卡**:引入 `qr-code-styling`(微信/支付寶設計語言:中心 logo + 品牌色 + 圓角碼點),React 台卡排版(桌號大字+店名+slogan+引導),導出**高 DPI PNG/SVG** 供印刷店製貼紙
- **訂單營銷碼·集字類**:每單唯一 token + `marketing_qr_tokens` 表(核銷即廢);與 `marketing_redemptions` 打通;掃碼核銷免手輸訂單號
- **加群類**:固定 URL,店主手動配置連結
- **驗收**:桌號碼可反覆印刷;偽造 token 被拒;舊碼寬限期內可用;集字碼掃碼核銷後二次掃提示已核銷;台卡導出可印刷;`cargo test` + 構建綠

### v2.9.0 — 掃碼商城增強(P0/P1)
- 營銷組件前後端對齊(`MarketingCard` 用後端 HTML,補齊 6 種)
- 商城加料/去料 + 規格數量 + 搜索 + 熱銷標記
- 掃碼首屏營銷鉤子(首掃運勢/領券)
- token 過期友善頁
- **驗收**:折價券等在手機端正確顯示;加料訂單正確入單;下架/不存在菜品被攔截

### v3.0.0 — 營銷閉環與數據(P1)
- 掃碼/轉化漏斗 + 組件效果分析報表

- **驗收**:漏斗數據準確;裂變券雙方到賬;報表查詢在可接受成本內(H)

## 8. 結論

三個方向中,**二維碼體系(方向 A)是樞紐**:桌號碼解決「設計感 + 印刷貼紙 + 防偽下單」,訂單營銷碼解決「每張唯一 + 掃碼核銷廢止」,同時把截圖兌獎升級為掃碼自動核銷,並為方向 C 的數據分析鋪設 token 基礎。建議 v2.8.0 優先落地 A,v2.9.0 補商城短板(尤其前後端營銷組件對齊這一低成本高價值項),v3.0.0 收口數據與裂變。

關鍵設計差異須牢記:**桌號碼固定**(實體貼紙不可輪換,靠簽名綁桌 + pending 閘門防護)、**訂單集字碼每單唯一可廢止**、**加群碼固定可手動換**——三類碼的生命週期與防護機制各不相同。

全程恪守項目約束:簡體中文界面、防呆零配置默認、面向小餐廳/飲品店、單台 Android PAD 可跑。
