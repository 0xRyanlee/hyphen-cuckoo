---

## 0. 方法說明

- **Kano 五分類**(站在小店店主/顧客視角):
  - **M 必備(Must-be)**:缺則不可用,有了不加分。出問題=災難。
  - **O 期望(One-dimensional)**:越好越滿意,線性。
  - **A 魅力(Attractive)**:超預期驚喜,缺了不失望,有了顯著加分。
  - **I 無差異(Indifferent)**:有沒有都無感,投入需謹慎。
  - **R 反向(Reverse)**:做了反而降低滿意(如強制會員、繁瑣配置)。
- **多維審計維度**:A 錯漏 / E 防呆 / F 對抗安全 / I 架構一致 / J 用戶旅程 / H 成本 / G 合規。

## 2. 多維審計(當前系統 v3.2.0)

| 維度 | 評估 | 發現 |
|---|---|---|
| **A 錯漏** | 良好 | 前後端抽字算法不一致(djb2 vs SipHash)已修(後端 picked_char 統一);**桌號存在性未校驗**(待) |
| **E 防呆** | 優 | 核銷 PIN 可選、零配置默認、下架校驗、IP 自動檢測+手動兜底、舊碼寬限期;多領無限制(權衡) |
| **F 對抗安全** | 中上 | HMAC 簽名防偽、redeem web 端 PIN、token 截斷 16B 常數時間比較;**殘留:桌號偽造可下單、campaign 多領刷券**(均已留檔權衡) |
| **I 架構一致** | 優 | secret/role-auth 本地文件慣例、新表走 init_tables、transport 雙模、token 模塊單一真相源 |
| **J 用戶旅程** | 優 | 掃碼→點單→彈窗→截圖→核銷閉環完整;fallback 重試;過期碼友善頁;三類碼生命週期清晰 |
| **H 成本** | 中 | index chunk 622KB(待路由級分包);vendor 已分;funnel 查詢有 30 天窗口+LIMIT,可控 |
| **G 合規** | 優 | 純截圖無會員 = 零 PII 收集,隱私友好;本地優先,無雲端數據外流 |

**審計總評**:架構一致性、用戶旅程、防呆、合規均優;**唯一 M 級缺口是桌號存在性校驗(A+F 維度)**;成本維度的 bundle 分包為 O 級優化。

## 4. 優先級建議(Kano × 審計交叉)

**P0(必做,M 級安全缺口)**
- **SEC1 桌號存在性校驗**:`create_self_order` 校驗 token/table_no 對應桌號存在於 `restaurant_tables`,堵偽造桌號刷單。唯一的 M 級待辦。

**P1(下一版 v3.3,O 級高價值)**
- C2 舊碼硬切開關(配合 SEC1 收口掃碼安全)
- MUL campaign 多領限制(促銷防刷,投放前必備)
- E1 路由級 lazy-loading(顧客掃碼首屏速度)
- B3' 完整 modifier(對齊 POS,結構化加料)

**P2(漸進增強)**
- B4 套餐、B7 折價券金額、B6 組件/時段分析(重度運營者)

**P3(暫緩,I 級低價值)**
- B8 告警、E3 health_check、E2 CSP —— 用戶無感,投入謹慎

**封存/規避**
- 🔒 E4 雲端多店、REF-A 雙向裂變、D5 跨單集字:均需突破「無會員/本地優先」的產品邊界,留待戰略決策
- 🚫 會員系統、強制 PIN:Kano 反向項,已正確規避,**不應再做**
